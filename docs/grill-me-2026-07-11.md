# 从 Electron+Java 回到 Rust: 头脑风暴 / 需求调研笔记
日期: 2026-07-11 · 目标: 重新评估 design.md 中"因 Rust 缺乏 JDBC 支持而选 Electron+Java"的前提，决定是否以及如何回到 Rust 开发

## 概述 / 核心决策
（动态更新）

**背景**：
- `design.md` (v3.0, 2026-07-08) 定位为"数据库结构管理工具"，JSON 为 SSOT，产出 DDL/ALTER/代码，永不自动执行。
- 技术栈为 Electron + Node(TS) 主进程 + Java 子进程(JDBC) 三层，打包 ~170MB（Electron 120 + JRE 40 + jar 10）。
- 选 Java JDBC 的唯一前提："Rust 对数据库缺乏类似 JDBC 的广泛支持"，尤其信创库（DM/KingBase/GBase）。
- 现看到 https://github.com/t8y2/dbx（Rust 写的 20MB 数据库客户端，支持 60+ 库含信创），前提被动摇，用户想回到 Rust。

**关键已验证事实**：dbx 对信创三库（DM/KingBase/GBase）也是 Java JDBC agent 连接，非纯 Rust。dbx 的"20MB"靠的是 Tauri 砍 Chromium + Java/JRE 按需下载，而非"Rust 连信创库"。详见下方调研结论。

**决策汇总（Q1-Q9 定稿）**：
1. **整体**：Tauri 2.x 桌面 + Rust 后端(`aqua-core`) + Vue3/element-plus 前端 + Java connector。推倒重写 core+Tauri,connector 复用,前端迁移。纯桌面(不要 Web/Docker)。
2. **连接层**：一次性命令模式(不常驻,低频设计工具);混合--MySQL/PG 走 Rust native(`mysql_async`/`tokio-postgres`)免 Java,Oracle/DM/KingBase/GBase/H2 走 Java JDBC;用户自备 JDK 17+,`connector.jar` 内置打包,JDBC 驱动+resolver jar 用户提供(`drivers/`+`registry.json`);不做 SSH,直连。
3. **反解**：Java 侧外置 jar(Q2 C 方案,URLClassLoader+registry.json);native(MySQL/PG)反解写死 Rust。两套分库归属。
4. **Driver 接口**：`trait Driver` 收敛(test/list_tables/get_columns/list_indexes/query_rows),native+Jdbc 两实现,返回统一 aqua schema 类型,反解藏各实现内。
5. **前端**：放弃 json-ui 用 element-plus;前端 JSON 生成(`generators/frontend-json`)保留,服务于外部 json-ui 项目。Tauri=Rust 后端+Web 前端(系统 webview)。
6. **CLI**：Tauri 二进制双模式(无 args 开 GUI/有 args 走 CLI),generate 逻辑在 `aqua-core`;generator 内置 4 个(ddl/java/json/strconst),扩展靠改程序。
7. **项目结构**：Cargo workspace(`crates/aqua-core`)+ `src-tauri`(Tauri 壳+CLI 入口)+ `app`(Vue 迁移)+ `connector`(Java 保留)+ `drivers/` + `registry.json`。
8. **体积**：安装包 ~20MB(Tauri+connector+前端,无 JRE),用 Java 库时用户自装 JDK 17+。原 design.md 170MB -> ~20MB。

## 调研结论（dbx 架构，2026-07-11 源码核实，main 分支 v0.5.53）

### A. 连接层：三类路径
1. **纯 Rust native（in-process，无 Java/无子进程）**：MySQL(mysql_async)、PG(tokio-postgres，含 GaussDB/openGauss)、SQLite(rusqlite)、DuckDB、SQLServer(tiberius)、Redis、Mongo、ClickHouse、ES、InfluxDB 等。
2. **Java JDBC agent（spawn `java -jar` 子进程）**：Oracle、**DM/KingBase/GBase 8a/8s**、Highgo、Vastbase、Goldendb、DB2、H2、Snowflake、Hive 等（共 39 个 driver 模块）。
3. **原生 agent（Go 二进制，无 JRE）**：仅 oracle-go、xugu。
- **GaussDB/openGauss -> native**（patched `tokio-postgres-gaussdb` fork，处理握手/认证差异）。
- **KingBase -> Java agent**（JDBC URL 参数 native tokio-postgres 解析不了，"dedicated agent metadata SQL"）。
- Highgo/Vastbase：连接走 agent，元数据回退 native postgres 池。

### B. Rust 调 Java 的机制
- `std::process::Command` spawn `java -jar <agent.jar>`（**非嵌入式 JVM**，无 jni-rs/graal）。
- 通信 = **stdin/stdout 换行分隔 JSON-RPC 2.0**，启动握手 `{"ready":true}`，协议 v1，28 个方法（connect/list_tables/get_columns/execute_query/...）。
- agent = **长驻守护进程**，单 JDBC Connection 复用，5s 健康检查自动重连，sessionId 游标分页。
- JVM 调优：`-XX:TieredStopAtLevel=1 -XX:+UseSerialGC`（轻量 sidecar）。
- Rust 侧 `AgentDriverClient`（`db/agent_driver.rs`）持有 `Child`+stdin/stdout，`AgentManager.daemons` HashMap 复用进程。

### C. 分发与打包
- 每个 agent = **shadowJar fat jar**，JDBC 驱动打进 jar 内（用户无需自备驱动，`external_driver_required:false`）。
- 驱动来源：多数 Maven Central；GBase 8a 用**本地 vendored jar**（`libs/gbase-connector-java-9.5.0.10-build1-bin.jar`，不上 Maven）。
- **JRE 21 + agent.jar 按需下载**（`agent-registry.json` from GitHub Releases，R2 CDN `dl.dbxio.com` 竞速 `race_download`，sha256 校验），落 `~/.dbx/agents/`，**不进安装包**。
- agent jar 体积：dameng 2.1MB / kingbase 1.6MB / gbase8a 1.3MB / gbase8s 3.4MB / snowflake 80.8MB / oracle 原生 ~18MB。
- `plugins/jdbc` = 独立可选 sidecar（自定义 JDBC URL，用户自备 Java+驱动，Maven resolver 拉驱动），与 `agents` 两套系统同协议。

### D. 核心抽象（dbx-core）
- **无统一 driver trait**，是 `PoolKind` 枚举 + 各驱动自由 async 函数 + `match` 分发的三段式隐式契约。
- native 与 agent 在**类型层**统一（都反序列化成 `types.rs` 的 `ColumnInfo`/`TableInfo`/`IndexInfo`/`QueryResult`），非 trait 层。
- 元数据：`schema.rs` 门面 + `match PoolKind` 分流；PG `get_columns` **三级 SQL fallback**（pg_catalog -> 兼容 -> information_schema）；公共 `ColumnInfo` 含 `full_type/numeric_precision/numeric_scale/character_maximum_length/is_pk/column_comment`。
- 全 async(tokio)；native 用各自池（deadpool-postgres / mysql_async::Pool），agent 单进程 `Arc<Mutex>`。

### E. 架构与打包
- **Tauri 2.x + Vue 3.5 + 系统 webview（无 Chromium）**。
- 通信：Tauri commands(invoke) + events；前端 `api.ts` 运行时按 `isTauriRuntime()` 在 invoke/HTTP 间切换。
- **20MB = 系统 webview（省 Chromium）+ Rust 极致 release profile（panic=abort+strip+lto+codegen-units=1+opt-level=s）+ Java/JRE/JDBC 全按需下载**。
- 三模式复用同一 `dbx-core`：Desktop(Tauri commands) / Docker(dbx-web axum HTTP/WS) / Web(dbx-web)。
- `dbx-web` = axum 后端 + 静态托管前端 dist + argon2 鉴权。
- CLI/MCP = 纯 JS/TS（`node-core` 自带 pg/mysql2 驱动），**不走 napi-rs**，可 HTTP 桥接 dbx-web 复用后端。

### F. 对 design.md 的核心启示
1. **信创库的 Java 依赖绕不开**（dbx 也没绕开），但可从"内置打包"改"按需下载"。
2. **真正的体积收益来自砍 Chromium（Tauri）+ Java 按需化**，而非"Rust 连信创库"。
3. 核心逻辑若换 Rust 需重写（design.md 当前是 TS）；**前端 Vue+json-ui 在 Tauri 下可保留**。
4. 元数据反解逻辑类型可借鉴 dbx 的多级 SQL fallback + 公共类型层。
5. dbx 无 trait 是因 60+ 库 + agent/native 混杂；本项目库少，**反而应用 `trait Driver` 收敛**编译期强制完整性。

### G. aqua 现有代码进度（~/work/aqua，2026-07-11 实读）
- pnpm workspace 三包：core(TS) / app(Electron) / connector(Java)，Trellis 管理任务，docs/database_schema_tool_detailed_design.md 为正式 design。
- **core (TS, ~2452 行)**：已实现 schema 模型(project/bizType/dataType/enum/table/field)、数据集(json/sqlite/coerce/validate/export/import)、连库导入(resolver/spawn/query)、generators(ddl 含 7 方言、java-entity、frontend-json、database-consts/StrConst)。**未见 diff/ALTER 生成器**。
- **connector (Java, ~1095 行)**：protocol(Request/Response/ErrorInfo/ErrorCode/DataSource)、jdbc(ConnectionFactory/MetadataReader/QueryExecutor)、registry(RegistryEntry/RegistryLoader)、command(Test/Query/Import)、Main。
- **app (Electron)**：electron.vite + src/main + src/preload + src/renderer，Vue3+element-plus+pinia+vue-router。**进度浅**: main 214 行(6 个 IPC: project:new/open/save/recent + dialog:open/save,逻辑=文件IO+调core+recent.json), renderer 546 行(Welcome+Editor 两视图)。**未用 json-ui**(package.json 无依赖,design.md 与实现不符,待澄清)。
- **resolvers/**：6 个外置 CJS JS（mysql/postgresql/oracle/dm/kingbase/gbase），core 用 require() 加载调 resolve(column)；H2 内置 TS。

### H. 连接层现状对比 dbx
| 维度 | aqua 现状 | dbx |
|---|---|---|
| 进程模式 | 一次性命令(spawn java -jar, stdin 一 JSON -> stdout 一 JSON -> exit) | 长驻 daemon |
| 协议 | 单次 JSON in/out | JSON-RPC 2.0 over stdio (id/handshake/会话) |
| 驱动 | 用户提供(drivers/ + URLClassLoader + registry.json) ✅符合用户设想 | agents 内置 fat jar / plugins/jdbc 用户提供 |
| JRE | design.md 内置精简 JRE(40MB) | 按需下载 |
| 反解逻辑 | core 侧(TS require 外置 JS resolver) | Java 侧(agent 内) |
| 连接复用 | 无(每次新 JVM ~1-2s 冷启动) | 有(单 Connection 复用) |

**关键**：aqua connector 已是"统一一个 + 用户提供驱动 + URLClassLoader"模式，符合用户"所有库走 JDBC、用户提供驱动"设想。主要差异：一次性 vs 长驻、反解在 core(TS) vs Java 侧。

## 问答日志

### Q1 - 连接层架构 + 反解归属
- 提问: 一次性命令 vs 长驻 daemon? 反解逻辑归谁?
- 记录:
  - **连接层保留一次性命令模式**(不常驻)。用户原话:"不赞成常驻,连接数据库是低频行为,我们是设计工具,这点与 dbx 不同"。JVM 冷启动在低频场景可接受。
  - **反解移到 Java 侧**。用户原话:"反解问题来自新增数据库支持,比如 oracle 反解 int 需要判断,以前外置 js 就是为了解耦。写死在代码中对扩展不利。没有更好办法就放到 java 端"。
- 标记: 反解在 Java 侧的组织方式(写死类 vs 外置 jar vs 外置脚本)待定 -> Q2

### Q2 - 反解在 Java 侧的组织方式
- 提问: A(写死类) / B(Groovy脚本) / C(外置jar+URLClassLoader) ?
- 记录: 用户选 **C**。原话:"C 很好"。复用现有 URLClassLoader + registry.json 机制(与加载 JDBC 驱动同一套),加库放 resolver jar 不动 connector 本体,零重编译无脚本引擎依赖。
- 修正: 用户随后提出部分库(MySQL/PG)走 Rust native 驱动免 Java -> 这些库无 Java 进程,反解不能在 Java 侧。C 方案仅适用于走 Java 的库。见 Q3。

### Q3 - 混合架构 + 反解分裂
- 提问: ①混合+反解分库归属 / ②反解统一Rust / ③全Java ?
- 记录: 用户选 **①**。MySQL/PG 走 Rust native 驱动(mysql_async/tokio-postgres)免 Java,反解写死 Rust 各驱动模块;Oracle/DM/KingBase/GBase/H2 走 Java JDBC,反解走 Q2 的 C 方案(Java 外置 jar)。用户只用 MySQL/PG 时零 Java 依赖。
- 代价: Rust 侧需为 MySQL/PG 实现元数据读取(list_tables/get_columns/list_indexes)+读数据行(query),参考 dbx mysql.rs/postgres.rs。两套反解机制并存(native 写死 + Java 外置)。
- 标记: Rust 侧 Driver trait 设计(native 与 JDBC 两实现统一接口)待定 -> Q4

### Q4 - Rust 侧 Driver 接口设计
- 提问: trait vs 枚举? async trait 方式?
- 记录: 用户**认可**推荐设计。`trait Driver`(test_connection/list_tables/get_columns/list_indexes/query_rows)收敛,两实现:`MySqlDriver`/`PostgresDriver`(native,反解写死 Rust)、`JdbcDriver`(spawn Java connector,收已反解逻辑类型)。返回统一 aqua schema 类型,反解藏各实现内。工厂 `create_driver(config) -> Box<dyn Driver>` 分发。native 一次性连接不维护池。
- 待定: async trait 用原生(Rust 1.75+,dyn 有限制)还是 `async-trait` crate(Box 开销)——若用 `Box<dyn Driver>` 动态分发则需 `async-trait`。实现期定。

### Q5 - 目标项目结构 + 迁移方式
- 提问: 纯桌面 vs 多模式? 推倒重写 vs 渐进?
- 记录: 用户定 **纯桌面**(不要 dbx-web/Docker/Web/axum 后端,aqua 是本地设计工具,无 Web 部署场景);**推倒重写** core+Tauri(现有未发布无运行压力,TS core 未全实现正好借机重做)。
- 目标结构: Cargo workspace(`crates/aqua-core` Rust 核心 + `crates/aqua-cli` Rust CLI)+ `src-tauri`(Tauri 壳)+ `app`(Vue/json-ui 迁移)+ `connector`(Java 保留复用)+ `drivers/` + `registry.json`。
- 迁移: aqua-core Rust 重写参考 TS 蓝本(不迁代码);src-tauri 替代 Electron 主进程;app 前端 IPC 改 Tauri invoke;connector 按 Q1-Q4 调整(反解移入外置 jar)。

### Q6 - JRE/connector 打包分发
- 提问: JRE 按需下载/内置/用户自备? connector 内置/下载?
- 记录:
  - connector.jar: **内置打包**(~10MB,含连接/元数据/反解框架,不含 JDBC 驱动)。
  - JRE: **用户自备 JDK 17+**(选③)。不内置不下载,安装包最小(~20MB)。用 Java 库(Oracle/信创/H2)前用户自装 JDK 17+;MySQL/PG 用户免 Java(Q3)。选 17(广泛 LTS)而非 21(dbx 用 21),connector 代码须兼容 Java 17 语法。
  - 驱动 + resolver jar: 用户提供(drivers/ + registry.json,Q2/Q3)。
- 衍生: app 需检测 JDK 存在且 >=17,缺失时提示用户安装(仅用 Java 库场景触发)。

### Q7 - json-ui 去留 + 前端 JSON 生成消费方
- 提问: json-ui 放弃/接入? frontend-json 生成消费方?
- 记录:
  - **放弃 json-ui,继续 element-plus**。用户原话:"放弃,原因是为了快,否则还要学 json-ui"。
  - **前端 JSON 生成(generators/frontend-json)必须保留移植**。用户原话:"必须要,这个工具的目的之一就是服务于使用 json-ui 的项目"。即 aqua 生成的前端 JSON 是给**使用 json-ui 的外部项目**消费,非 aqua 自身 UI。
  - 用户原话:"我以为 dbx 使用 rust 开发的前端"。-> **误解,已澄清**:dbx 前端是 Vue3(Web 技术),Rust 仅后端核心(dbx-core)+ Tauri 壳(src-tauri)。Tauri 架构下前端必然是 Web 技术(Vue/React/Svelte),系统 webview 加载。无"Rust 写前端";纯 Rust GUI(egui/slint/dioxus)是另一条路,放弃 Web 生态。用户选 element-plus 即接受 Web 前端,aqua = Rust 后端 + Vue 前端,与 dbx 一致。用户回复"明白了"确认,继续 Vue+element-plus 路线(不走纯 Rust GUI)。

### Q8 - CLI 形态 + generator plugin
- 提问: CLI 独立二进制/Tauri模式/两者? generator 内置/动态plugin?
- 记录:
  - CLI 选 **B(Tauri app 二进制 CLI 模式)**。用户原话:"选B,因为开发环境用,装一个就行"。一个二进制无 args 开 GUI、有 args(如 `aqua generate`)走 CLI 不开窗。
  - **修正 Q5 结构**:去掉 `crates/aqua-cli`,CLI 入口在 `src-tauri/main.rs`(args 检测),generate 逻辑在 `aqua-core`。
  - generator **第一版内置 4 个(ddl/java/json/strconst)硬编码**。用户原话:"内置,以后增加就修改程序"。plugin 机制延后,扩展靠改程序重编译。

### Q9 - SSH 隧道
- 提问: 信创库内网需SSH? 第一版做不做?
- 记录: 用户选 **②不做,直连**。第一版不做 SSH 隧道,用户自保证库可达(VPN/直连/端口转发)。数据源配置保持 type/host/port/database/user/password。

### Q10 - 收尾表态
- 自动更新: **不做**(第一版手动下载更新)
- i18n: **仅中文**
- 移植起点(用户让定): **schema**(数据模型基础,所有 generator/dataset/import/diff 依赖)-> generators(纯逻辑快速验证) -> dataset -> import(依赖 Driver trait) -> diff/ALTER(新功能)

### Q11 - 开发目录
- 提问: 原地重写 vs 新建? 目录名?
- 记录: 用户选 **A: 新建 ~/work/aqua,原目录 mv 为 ~/work/aqua-legacy**。理由:推倒重写不混新旧;原 TS core 作参考蓝本;Trellis 重新 init;connector 拷贝到新目录继续开发。
